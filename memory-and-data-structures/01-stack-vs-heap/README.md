# 01. Stack vs Heap (Статична vs Динамічна пам'ять)

Кожен раз, коли ви створюєте змінну, масив чи функцію, рушій V8 має виділити (allocate) фізичне місце у вашій оперативній пам'яті (RAM). Але куди саме потраплять ці дані? JavaScript-рушій автоматично сортує їх у два гігантські "сховища": **Стек (Stack)** та **Купу (Heap)**.

Розуміння цієї різниці є ключем до того, як JavaScript передає дані між функціями і чому зміна одного об'єкта може несподівано зламати інший (мутації).

---

## I. Stack Memory (Статична пам'ять)

**Теза:** Stack — це дуже швидка, лінійна, але суворо обмежена в розмірах ділянка пам'яті. Вона ідеальна для зберігання даних, розмір яких комп'ютер знає ДО того, як запустить код. Це місцевість **примітивів**.

### Приклад
```javascript
let age = 30;
const isDeveloper = true;
let codeLabel = "A";
```

### Просте пояснення (Junior/Middle)
Уявіть Stack як стопку коробок однакового розміру. Коли ви пишете код, двигун знає: число (`number`) завжди займає рівно 8 байтів (64 біти). Булеан (`boolean`) займає ще менше. Оскільки ці коробки стандартного розміру, двигун просто складає їх одну на одну і миттєво до них звертається. Це супершвидко, але коробки не можуть магічним чином розширитися під час роботи.

До примітивів у JS належать: `String`, `Number`, `Boolean`, `Undefined`, `Null`, `Symbol`, `BigInt`. Усі їхні **значення зберігаються безпосередньо у Стеку**.

### Технічне пояснення (Senior)
Стек прив'язаний до **Execution Context** (про який ми детально говорили в попередньому блоці). Кожен раз, коли викликається функція, виділяється новий *Stack Frame* (фрейм стека). В межах цього фрейму в `Environment Record` створюються змінні. 
Оскільки апаратний стек процесора не підтримує динамічне розширення і працює за принципом LIFO (Last-In-First-Out), він фізично не може зберігати масиви чи об'єкти, до яких ви можете будь-якої миті додати 10 нових властивостей. Оптимізатор V8 зберігає маленькі цілі числа (SMI - Small Integers), такі як 42, безпосередньо всередині самого адресного покажчика (pointer tagging), не виділяючи під них навіть окремої пам'яті!

### Edge Cases / Підводні камені
> [!WARNING]
> Оскільки Stack суворо обмежений (в V8 це зазвичай ~1MB-10MB залежно від системи), безкінченна рекурсія миттєво його переповнить (**Maximum call stack size exceeded**). Проте лише кількість функціональних фреймів є ризиком: самі по собі примітиви майже ніколи не викличуть переповнення стеку під час звичайної роботи програми.

## II. Heap Memory (Динамічна пам'ять)

**Теза:** Heap (Купа) — це гігантська, неструктурована область пам'яті (як велике звалище). Вона призначена для даних, розмір яких може динамічно зростати або зменшуватися. Це місцевість **Reference Types** (Об'єктів, Масивів, Функцій).

### Приклад
```javascript
const user = {
    name: "Artur",
    role: "Admin"
};

const permissions = ["READ", "WRITE"];
```

### Просте пояснення (Junior/Middle)
Ви не можете запакувати великого слона в стандартну коробку зі Стека. Якщо ви створили об'єкт `user` і завтра додасте туди ще 100 полів `user.address`, `user.phone` і т.д., коробка порветься.
Тому рушій кладе слона на величезну галявину (Heap). На цій галявині можна будувати і добудовувати все, що завгодно. Але щоб знайти слона на галявині, вам потрібна мапа (вказівник/адреса).

### Технічне пояснення (Senior)
Allocation (виділення пам'яті) в Heap відбувається повільніше через механізми пошуку вільних блоків RAM (зокрема через Fragmentations). Після того як V8 кладе об'єкт у пам'ять, йому призначається 64-бітна унікальна адреса в пам'яті, наприклад `0x1034f40d2`.
Сам Heap розділений на кілька поколінь (Young Generation / Old Generation) для ефективнішого збирання сміття (Garbage Collection). Майже всі нові об'єкти потрапляють у Nursery (Young Generation), і лише якщо вони переживають кілька циклів прибирання, їх переносять в Old Space, де їх виділяти і знищувати ще важче.

### Edge Cases / Підводні камені
> [!CAUTION]
> **Memory Leaks (Витоки пам'яті)**
> Збирач сміття (Garbage Collector) звільняє пам'ять у Heap лише тоді, коли на об'єкт не залишилося жодного посилання з активних коренів (алгоритм Mark-and-Sweep). Якщо ви випадково збережете глобальне посилання на об'єкт, який більше не потрібен (наприклад, забутий `eventListener` чи ключ у глобальному Map), об'єкт назавжди залишиться у Heap і викликатиме витік пам'яті.

## III. Процес: Як Stack і Heap працюють разом

**Теза:** Змінні у Stack **ніколи** не зберігають сам об'єкт чи масив. Вони зберігають лише **Pointer (Вказівник/Адресу)** на те місце в Heap, де фізично лежить об'єкт.

### Візуалізація
> [!TIP]
> **[▶ Запустити інтерактивний візуалізатор (Stack vs Heap allocation)](../../visualisation/memory-and-data-structures/01-stack-vs-heap/memory-layout/index.html)**

### Процес крок за кроком:
Коли ви створюєте `const user = { name: "Artur" }`:
1. Розпочинається розбір коду рушієм і запис в Heap. 
2. V8 шукає в Heap вільні байти і створює там структуру (хеш-таблицю або Hidden Class) `{"name": "Artur"}`.
3. V8 повертає нам адресу (pointer), де він поклав цей об'єкт: наприклад, `<0xAD42>`.
4. В `Execution Context` (у Стеку) створюється змінна `user`, і їй як "значення" записується ця адреса `<0xAD42>`.

Тобто `user` у стеку важить якихось 8 байтів (розмір адреси), але сам об'єкт у Heap може займати мегабайти!

### Edge Cases / Підводні камені

> [!WARNING]
> **Примітиви не копіюють адресу! Вони копіюють значення (Pass by Value).**
> ```javascript
> let stackAge = 30;
> let copyAge = stackAge;
> copyAge = 40; // stackAge залишиться 30
> ```
> В цьому прикладі `copyAge` отримав абсолютно нову "коробку" у Стеку зі значенням 30. Кожна змінна-примітив 100% унікальна і безпечна.

> [!CAUTION]
> **Об'єкти копіюють АДРЕСУ в Стеку (Pass by Reference).**
> ```javascript
> const heapUser = { age: 30 };
> const copyUser = heapUser;
> 
> copyUser.age = 40; 
> console.log(heapUser.age); // 40 ! 
> ```
> Чому це сталося? Згадаймо: `heapUser` тримав у стеку адресу `<0xAD42>`. Рядок `copyUser = heapUser` зробив лише одне: він скопіював цю адресу `<0xAD42>`, створивши нову "коробку" у Стеку.
> Але **ОБИДВІ змінні тепер вказують на ОДИН І ТОЙ САМИЙ об'єкт**, який самотньо лежить у Heap. 

Зміна даних за спільним посиланням (Pointer) називається **Мутацією (Mutation)** і є причиною половини всіх логічних багів у JS. Про те, як керувати цілком дозволеними мутаціями (The Const Paradox), ми поговоримо у наступній статті!
