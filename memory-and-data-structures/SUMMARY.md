# Memory and Data Structures: Module Summary

Вітаємо! Ви пройшли один з найскладніших модулів — архітектуру пам'яті JavaScript та роботу двигуна V8. Тут зібрана вижимка найкращих практик, аналогій та переліків "червоних прапорців", які допоможуть уникнути падіння продуктивності або провалу на співбесіді рівня Senior.

## 🏗️ Mental Models (Словник-Асоціація)

| Концепція | Ментальна Аналогія | Що Вона Означає? |
| :--- | :--- | :--- |
| **Stack** | 📦 Стопка коробок (фіксований розмір) | Швидка LIFO-пам'ять для примітивів та виконання коду (Execution Contexts). Коли функція завершується, коробку викидають (якщо немає замикання). |
| **Heap** | 🏭 Величезний склад | Неструктуроване місце для життя об'єктів (Objects, Functions, Arrays). Ви не можете покласти об'єкт у коробку, ви кладете туди тільки шматок паперу з **адресою** складу (pointer). |
| **Let/Const TDZ** | 🚧 Стрічка "Поліція, не перетинати" | Змінна вже існує (хоіститься), але до неї заборонено торкатися до рядка її ініціалізації. |
| **Lexical Environment**| 📒 Журнал + Ключ від дверей | Журнал записує ваші змінні (Environment Record). Ключ — це `[[OuterEnv]]`, який відчиняє двері до журналу батька. |
| **The "Const" Paradox**| 🔒 Суперклей для вказівника | `const` приклеює ДО СТЕКУ (Stack) адресу складу. Ви не зміните адресу на інший склад, але ви можете піти на склад і рознести там усе (мутація Heap-об'єкта). |
| **V8 Hidden Classes** | 🚙 Конвеєр машинобудування | Якщо ви збираєте машини (об'єкти) завжди з однаковим порядком деталей, V8 працює як конвеєр — миттєво (`Monomorphism`). Якщо постійно ліпити деталі хаотично (`delete`), двигун розгубиться і збиратиме кожну машину вручну (`Dictionary Mode`). |
| **Garbage Collection (GC)** | ⚡ Струм по гірлянді | GC Roots — це розетка. Якщо струм (посилання) не доходить до об'єкта в Heap, світло гасне, і GC змітає його в смітник (Mark-and-Sweep). |

## 🔑 Key Points (Золоті правила V8)

1. **Немає магії, є лише Lexical Environment та Heap:** Завжди уявляйте змінну як два кроки: пошук ідентифікатора в Словнику середовища, і перехід за його адресою у Купу (якщо це об'єкт).
2. **"Delete" — це ворог №1:** Ніколи не вживайте `delete obj.prop`. Це ламає "Shape" (дерево переходів V8) об'єкта, скидаючи його оптимізацію і перетворюючи на повільну хеш-таблицю. Використовуйте `obj.prop = undefined` або `null`.
3. **Monomorphism & Arrays:** Усі об'єкти, які проходять через одну функцію, мають містити властивості в **однаковому порядку**! Те саме з масивами: `PACKED_SMI` (лише цілі числа: `[1, 2, 3]`) — найшвидші. Але як тільки ви робите "дірку" (`[1, , 3]`), масив спадає до `HOLEY_ELEMENTS`, руйнуючи швидкість циклу в 10 разів.
4. **Shallow vs Deep Copy:** Spread-оператор `{...obj}` — це `Shallow Copy` на стеку. Він створює новий об'єкт, але копіює старі АДРЕСИ (Shared Mutable State) всіх вкладених полів. Щоб відірватися на 100%, потрібен `structuredClone(obj)`.

## 🛠️ Memory Health Checklist (Як знайти витік пам'яті)

Перед релізом великого компонента задайте собі питання:
- [ ] Чи всі `setInterval` або `requestAnimationFrame` мають відповідний `clear...()` у блоці unmount/cleanup?
- [ ] Чи знято `.removeEventListener()` із глобальних об'єктів `window` / `document` після відпрацювання логіки компонента? (Якщо не зняти, EventListener стане GC Root "Якорем" і заблокує видалення всього контексту у Heap).
- [ ] Чи є у мене "важкі" об'єкти (зображення, Base64 стрічки), які висять у довготривалому замиканні (Closure), хоча більше не потрібні?
- [ ] Чи не залишив я змінні без ключових слів `let/const`? (Вони тихо прив'язуються до глобального `window`).

## 🌶️ Interview Questions (Gotchas)

**Q. Що виведе цей код і чому?**
```javascript
const a = { val: 1 };
const b = a;
a.val = 2;
console.log(b.val);
```
> **A.** Виведе `2`. Змінна `a` і `b` знаходяться в Стеку, проте вони зберігають одне й те саме посилання на об'єкт `{ val: 1 }`, який знаходиться в Heap-пам'яті (Shared State). Зміна за посиланням `a` мутує фізичний об'єкт у Купі. Відповідно, звернення до того самого об'єкта через змінну `b` покаже мутоване значення.

**Q. Який із двох об'єктів працюватиме швидше в циклах і чому? `const o1 = {x: 1, y: 2}; const o2 = {y: 2, x: 1};`**
> **A.** Вони працюватимуть повільно, якщо передавати їх в ОДНУ функцію. Об'єкти мають різний порядок ключів, тому під капотом V8 створює для них різні `Hidden Classes (Shapes)`. Функція, що їх приймає, стане `Polymorphic` (відключиться Inline Caches), втративши можливість звертатися до полів за закешованим зміщенням (Offset) в пам'яті за швидкістю O(1).

**Q. Що станеться, якщо зробити `let x = 10;`, потім `delete x;`, і які наслідки оператора `delete` для об'єктів?**
> **A.** Оператор `delete` не працює на локальних змінних (створених через `let/const/var`), він поверне `false`. Якщо застосувати його до полів об'єкта (`delete obj.prop`), він спрацює, АЛЕ це повністю зруйнує Hidden Class (Shape) цього об'єкта у V8. Двигун переведе об'єкт з режиму 'Швидких властивостей' до режиму повільної Хеш-таблиці (Dictionary Mode), через що будь-який подальший доступ до полів цього об'єкта впаде у швидкості. Правильний шлях: `obj.prop = undefined`.

**Q. Чому використання `let x = new Array(100)` — дуже погана ідея з точки зору V8 двигуна?**
> **A.** Ця операція створює об'єкт типу `HOLEY_ELEMENTS` (масив з сотнею дірок `empty`). Коли ви намагатиметеся прочитати чи перебрати порожню комірку `x[5]`, замість прямого доступу V8 доведеться піднятися по ланцюжку прототипів до `Array.prototype`, щоб переконатися, що властивості '5' там дійсно немає. Це ламає продуктивність у 10 разів порівняно зі звичайним `PACKED_SMI` масивом `[1, 2, 3]`. Набагато швидше ініціалізувати: `new Array(100).fill(0)`.

**Q. Коли Garbage Collector (GC) видалить змінну, оголошену всередині функції?**
> **A.** За замовчуванням: щойно функція закінчить своє виконання, її Lexical Environment буде видалено зі стеку (Stack Frame Popped). Оскільки на змінні функції більше немає `Global Roots`, під час наступної фази `Mark-and-Sweep` вони будуть знищені. Але якщо всередині повернулася інша функція (Closure), яка посилається на цю змінну — Lexical Environment батька залишається жити в Heap нескінченно довго (доки існує посилання на дочірню функцію).
