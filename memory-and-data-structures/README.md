# Block II: Memory & Data Structures (Управління пам'яттю)

Вітаємо у другому великому блоці JS-Lab! Якщо в першому розділі ми розібрали, **як** виконується код (Execution Context, Call Stack), то тут ми зануримося у питання: **де** він виконується і як дані фізично розміщуються у "залізі".

Автоматичне керування пам'яттю (Garbage Collection) в JavaScript часто створює ілюзію, що пам'ять — це нескінченний та безкоштовний ресурс. Але для Senior-розробника розуміння RAM — це база, яка дозволяє писати код, що не просто "працює", а працює максимально швидко.

---

## 1. Memory Management: Від магії до контролю
Багато хто вважає, що GC звільняє розробника від думки про пам'ять. Це помилка. 
**Теза:** Знання того, як V8 виділяє (allocate) та звільняє пам'ять, дозволяє уникати "витоків" (Memory Leaks) та писати код, який не змушує рушій "гальмувати" під час прибирання сміття.

---

## 2. The "Hardware" Perspective
З точки дії рушія, ваша оперативна пам'ять — це обмежений ресурс. 
- **Stack:** Швидка, але маленька ділянка для миттєвих операцій.
- **Heap:** Велика "куча", де дані можуть жити довго, але доступ до них повільніший, а прибирання — складніше.
JavaScript-розробник має розуміти цей баланс, щоб не перевантажувати купу (Heap) непотрібними об'єктами.

## 4. Глосарій термінів (To Learn)

| Термін | Коротке визначення |
| :--- | :--- |
| **Pointer** (Вказівник) | Адреса в пам'яті, що вказує на реальні дані в Heap. |
| **Allocation** | Процес виділення місця в RAM під нову змінну чи об'єкт. |
| **Monomorphism** | Стан, коли функція завжди отримує об'єкти однієї "форми" (Hidden Class). |
| **Deoptimization** | Відкат рушія до повільного виконання через зміну структури даних. |
| **Reachability** | Критерій GC: об'єкт живе, поки до нього можна "дійти" від корня (Root). |
